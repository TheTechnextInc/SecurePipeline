1. PROJECT OVERVIEW
================================================================================

This platform is a fully interactive DevSecOps (Development + Security +
Operations) dashboard that visualizes CI/CD pipeline stages, shows security
checks from SonarQube, highlights failed security gates, and provides
comprehensive security insights.

It is designed to demonstrate a real-world DevSecOps workflow for:
  - Monitoring CI/CD pipelines in real-time
  - Tracking security vulnerabilities (SAST/DAST findings)
  - Enforcing quality gates that block insecure deployments
  - Generating compliance and security reports
  - Managing integrations (GitHub, SonarQube, Webhooks)

The app uses LOCAL COOKIE-BASED AUTHENTICATION with pre-configured demo
accounts. All dashboard pages are protected - you must log in to access them.

DEFAULT LOGIN CREDENTIALS:
  Email:    admin@securepipe.dev
  Password: Admin@123456

  OR

  Email:    demo@securepipe.dev
  Password: Demo@123456

2. ARCHITECTURE & TECHNOLOGY STACK
================================================================================

Frontend Framework:   Next.js 16 (App Router)
Language:             TypeScript
Styling:              Tailwind CSS v4 with shadcn/ui components
Charts:               Recharts (AreaChart, BarChart, LineChart, PieChart)
Icons:                Lucide React
Authentication:       Local cookie-based auth (no external services)
State Management:     React Context API (centralized global store)
Middleware:           Next.js middleware for route protection

Key Architectural Decisions:
  - All state is centralized in /lib/store.tsx using React Context
  - Any action on any page (add repo, run pipeline, fix vulnerability)
    immediately reflects across ALL other pages
  - Authentication is handled via secure HTTP-only cookies
  - Middleware protects all routes except /auth/* pages
  - No localStorage is used - state lives in React Context
  - Demo data is loaded on first render to showcase the platform
  - NO EXTERNAL SERVICES REQUIRED - works fully offline/standalone

4. AUTHENTICATION FLOW (Local Cookie-Based Auth)
================================================================================

The authentication system uses LOCAL COOKIE-BASED AUTHENTICATION with
pre-configured demo accounts. No external services are required.

DEFAULT ACCOUNTS:
  1. Admin: admin@securepipe.dev / Admin@123456
  2. Demo:  demo@securepipe.dev / Demo@123456

FLOW:

  1. User visits any page (e.g., /)
  2. middleware.ts intercepts the request
  3. middleware.ts calls getSession() from /lib/auth.ts
  4. getSession() reads the securepipe_session cookie
     - If cookie exists and is valid: allows access
     - If no cookie or invalid: redirects to /auth/login

  5. On /auth/login, user enters email and password
  6. Form submits to server action login() in /app/auth/actions.ts
  7. Server action calls authLogin() from /lib/auth.ts which:
     - Validates credentials against DEFAULT_USERS array
     - On success: sets HTTP-only cookie with user data, redirects to /
     - On error: redirects to /auth/error?message=...

  8. Sign out: sidebar has sign out button that calls signout() server action
      - Deletes the auth cookie
      - Redirects to /auth/login

FILES INVOLVED:
  - middleware.ts          -> Entry point for all requests
  - lib/auth.ts            -> Core auth logic (login, logout, getSession)
  - app/auth/actions.ts    -> Server actions for login, signout
  - app/auth/login/page.tsx -> Login form UI with pre-filled credentials

COOKIE DETAILS:
  - Name: securepipe_session
  - HttpOnly: true (prevents XSS attacks)
  - Secure: true in production
  - SameSite: lax
  - MaxAge: 7 days

5. GLOBAL STATE MANAGEMENT (React Context Store)
================================================================================

All application state lives in /lib/store.tsx using React Context.

The StoreProvider wraps the entire app in layout.tsx. Every page that needs
data calls useStore() to access the shared state and actions.

STATE INCLUDES:
  - repositories[]         -> Connected GitHub repos
  - pipelines[]            -> CI/CD pipeline configurations and runs
  - vulnerabilities[]      -> Security findings from SAST scans
  - qualityGates[]         -> Quality gate definitions and conditions
  - notifications[]        -> System alerts and notifications
  - reports[]              -> Generated and scheduled reports
  - webhooks[]             -> Configured webhook endpoints
  - sonarConfig            -> SonarQube connection settings
  - selectedRepo           -> Current repository filter (global)
  - searchQuery            -> Global search query
  - activeSettingsTab      -> Currently active settings tab
  - refreshing             -> Loading state for refresh
  - rescanning             -> Loading state for security rescan

ACTIONS (functions that modify state):
  - addRepository()        -> Adds a GitHub repo + auto-populates demo data
  - removeRepository()     -> Removes repo + cleans up its pipelines
  - addPipeline()          -> Creates new pipeline for a repo
  - runPipeline()          -> Simulates running pipeline with animated stages
  - removePipeline()       -> Deletes a pipeline
  - updateVulnerabilityStatus() -> Mark vuln as fixed/ignored/reopened
  - triggerRescan()        -> 3-second scan simulation, adds new finding
  - addQualityGate()       -> Creates custom quality gate
  - addCondition()         -> Adds condition to a gate with random actual value
  - removeCondition()      -> Removes condition from gate
  - setDefaultGate()       -> Sets a gate as the default
  - markAsRead()           -> Marks notification as read
  - markAllAsRead()        -> Marks all notifications as read
  - deleteNotification()   -> Removes a notification
  - scheduleReport()       -> Generates report with 3s loading animation
  - downloadReport()       -> Downloads report as text file
  - addWebhook()           -> Adds webhook endpoint
  - toggleWebhook()        -> Enables/disables webhook
  - testSonarConnection()  -> Simulates connection test with loading state

CROSS-PAGE REACTIVITY:
  - When you add a repo in Settings, it appears in Pipeline's repo dropdown
  - When you run a pipeline, a notification appears in Notifications
  - When you fix a vulnerability, the overview stats update immediately
  - When you change the repo filter in the header, ALL pages filter their data
  - When you create a report, the report count on Reports page updates

7. SONARQUBE API INTEGRATION (Simulated)
================================================================================

The platform SIMULATES SonarQube integration for demonstration purposes.

In Settings → Integrations → SonarQube, you can:
  - Enter a SonarQube server URL
  - Enter an authentication token
  - Click "Test Connection" to simulate connection testing

The SonarQube metrics shown on the dashboard (Bugs, Vulnerabilities,
Code Smells, Security Hotspots) are CALCULATED from the vulnerabilities
array in the store, not fetched from a real SonarQube server.

METRICS CALCULATION:
  - Bugs = Count of "Injection" or "XSS" type vulnerabilities
  - Vulnerabilities = Count of open critical/high severity vulns
  - Code Smells = Count of "Security Misconfiguration" type
  - Security Hotspots = Count of "Cryptography" or "Secrets" type

8. HOW TO ADD YOUR OWN GITHUB REPOSITORY
================================================================================

1. Navigate to Settings page (/settings)
2. Click "Add Repository" button in the GitHub section
3. Enter your repository name in format: "username/repo-name"
4. Click "Add Repository"

When you add a repository:
  - It appears in the connected repositories list
  - A new pipeline is automatically created for it
  - Sample vulnerabilities are added for demonstration
  - The repo appears in the header filter dropdown
  - All pages can now filter data by this repo

10. DEVSECOPS WORKFLOW EXPLAINED
================================================================================

This platform demonstrates a typical DevSecOps workflow:

  [Developer Commits Code]
           ↓
  [CI/CD Pipeline Triggered]
           ↓
  [Stage 1: Source] → Clone repository
           ↓
  [Stage 2: Build] → Compile/bundle application
           ↓
  [Stage 3: SAST Scan] → Static security analysis (SonarQube)
           ↓
  [Stage 4: Security Gate] → Check for vulnerabilities
           ↓
  [DECISION POINT]
     |
     +-- If critical/high vulnerabilities found → BLOCK DEPLOYMENT
     |
     +-- If no vulnerabilities → Continue to deploy
           ↓
  [Stage 5: Deploy] → Deploy to production

The key principle: Security issues BLOCK deployments automatically.
This prevents vulnerable code from reaching production.
